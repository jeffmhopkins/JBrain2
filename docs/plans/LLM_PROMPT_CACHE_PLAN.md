# Local LLM prompt cache — cache-stable prompt layout + gateway KV reuse

> **Status:** Scheduled · **Last verified:** 2026-07-16 · **Waves:** W1◻️ W2◻️

Cut the first-token latency of on-box turns by making the large, static jerv /
curator system prompts **reusable** across turns instead of re-prefilled every
call. Two independent waves: a backend change that keeps the prompt **prefix
byte-stable** (so llama.cpp can reuse its KV), and a gateway change that turns on
llama-server's **prompt-cache / prefix-reuse** flags. Peer to
`../reference/PROCESS.md` (the wave loop) and `../reference/DEVELOPMENT.md`
(standards); the on-box measurement path is the owner debug console
(`../runbooks/DEBUG_ACCESS.md`).

## Why the first token is slow

The slowness owners see as "initial load" is **prompt prefill** — the model
processing the whole system prompt + history before it emits token one — not the
model's cold weight-load. On a 122B that prefill is seconds-to-tens-of-seconds,
and it repeats every turn because nothing is reused. Grounded in the code:

- **The system prompts are already cache-friendly.** `jerv` and the `curator`
  persona are static `.prompt` files (`backend/src/jbrain/agent/prompts/jerv.prompt`
  ≈3–4k tokens, `system.prompt` ≈1.5k), rendered once at import with **no template
  vars** (`agent/agents.py:189`, `agent/loop.py:72`), and passed verbatim as the
  system message (`api/agent.py:681`). Their leading tokens are identical
  turn-over-turn — the ideal cache prefix.
- **A volatile block breaks the prefix.** The turn's current date+time
  (`now_block`, `agent/clock.py:48`) is prepended as the **first message after the
  system prompt** (`api/agent.py:593`; same in `tasks/runner.py:176`,
  `agent/spawn.py:826`). Because it changes every minute, only the system prompt is
  reusable — the entire growing conversation history that follows it is invalidated
  every turn.
- **The gateway keeps no prefix.** `llm/llama_swap_config.py` (`render`, ~`:93`)
  builds `llama-server` with **one slot and no cache flags** — no `--cache-reuse`,
  no `-np/--parallel`, no `--slot-save-path`. One slot retains only the
  most-recent prompt's KV, and without `--cache-reuse` a stable prefix isn't even
  salvaged once a later token diverges.
- **The slot gets clobbered.** Any local model can be the target of many of the
  ~25 routable tasks (`router.py` `TASK_DEFAULTS`, ~`:44`), each with a different
  system prompt. Interleaving them on one slot evicts the previous task's KV, so a
  jerv turn sandwiched between background tasks re-prefills its persona from cold.

## Wave W1 — Cache-stable prompt prefix (backend)

Make `[system prompt + conversation history]` byte-stable so llama.cpp reuses it
and prefills only the newest turn.

- **Task W1.1 — Move the volatile blocks to the tail.** Relocate `now_block`
  (and the `presence` / `me` blocks) from the head of the message list to
  **immediately before the latest user turn**, so the stable prefix grows
  append-only. Touch the three assemblers that share the pattern:
  `api/agent.py:573/584/593`, `tasks/runner.py:176`, `agent/spawn.py:826`. Keep
  the model's access to the current time — it still sees `now_block`, just at the
  tail. Confirm no prompt/tool copy assumes the time sits near the top.
- **Tests.** Unit-assert the assembled message order for each of the three paths:
  the volatile block is the last context message before the newest user message,
  and the system message + prior history are unchanged. LLM calls faked.
- **Verification.** On-box, before/after: two consecutive identical-prefix jerv
  turns; confirm the second turn's prefill (first-token latency / prefill tokens)
  drops. Measured via `POST /api/debug/complete` timing + the gateway prefill log
  (`GET /api/debug/llm/gateway-logs`).

## Wave W2 — Gateway prompt-cache flags (gateway)

Turn on llama-server's own prefix reuse so a stable prefix survives small
divergences and, optionally, model reloads.

- **Task W2.1 — Emit cache flags.** Add `--cache-reuse N` to the command in
  `llm/llama_swap_config.py` `render` (`:93`), and **evaluate** `--slot-save-path`
  (persist the jerv prefix across a cold model swap) and `-np`/`--parallel` (a
  second slot so an interactive prompt keeps its KV beside a background one) —
  guarded by their memory cost on the 128 GB box. Config is regenerated by
  `render`/`write` and hot-reloaded via `--watch-config`; a flag change needs a
  clean model relaunch (see the ops note below), not just a config swap.
- **Tests.** `render()` unit test asserts the chosen flags appear in the emitted
  command for a model that opts in; unaffected models are unchanged.
- **Verification.** On-box: repeat a jerv prompt with a one-token change mid-body;
  confirm `--cache-reuse` salvages the shared prefix (lower prefill) versus the
  W1-only baseline.

## Verification methodology

Both waves are perf changes with no unit-testable timing, so each carries an
**on-box measurement** through the debug console (`../runbooks/DEBUG_ACCESS.md`),
not just green unit tests: baseline first-token latency / prefill-token count,
the change, then the same measurement. Record the numbers in the wave PR. Ops
caveat learned live: changing a model's serving flags requires **unloading the
model** so llama-server relaunches with the new command — a `--watch-config`
hot-reload alone can leave the running process wedged.

## Complementary operational lever (not a code wave)

Routing the frequent **interactive** tasks (`agent.turn`, `agent.vision`) to a
model whose slot the **background** pipeline tasks (`note.extract`, `wiki.*`,
`fact.adjudicate`, pet tasks) don't share keeps the interactive prefix warm
between turns. This needs no code — `router.py` per-task overrides already allow
it — so it's an operator setting the plan recommends alongside the code waves,
not a wave itself.

## Open decisions

- **W1 placement.** Volatile block as the last *user* message vs a trailing
  context message just before the newest turn — pick the layout that both caches
  and keeps the model's time-awareness intact.
- **W2 flag set.** `--cache-reuse` value; whether `--slot-save-path` disk
  persistence earns its keep; whether a second slot (`-np 2`) fits the memory
  budget beside a 74 GB model. Confirm the box's llama.cpp build accepts each flag
  before committing (an unknown flag fails llama-server at load).

## Non-goals

- **Cloud prompt caching** (Anthropic `cache_control`) — a separate, provider-side
  mechanism; the adapter carries none today and this plan does not add it.
- **The jpet wall-pet "brain" prompts** (`jpet/brain.py`) — volatile-first by
  design (name/mood/objects/memories interpolated at the head) and single-shot, so
  prefix caching can't help them; explicitly out of scope.
