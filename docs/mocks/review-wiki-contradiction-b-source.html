Why this design · outcome mapping

The stored card is only {entity_ids:[A,B], summary} — not rulable on a phone. This screen re-hydrates it and leads with the raw source: the exact MyChart list both entities were extracted from, with the two flagged rows tinted (steel = A, amber = B) and every other appointment kept visible but greyed.

Grounding beats argument. Once you see two distinct rows in one screenshot, the false positive is self-evident — no cross-referencing required. Each entity's extracted facts hang directly under the line they came from, so provenance is physical: fact → source line.

The three rulings map straight to backend outcomes:

Dismiss→ resolve card, keep A & B distinct
Supersede→ retire loser, retain winner's values
Correct→ file correction note (channel #7)

The wiki is machine-written — a human never hand-edits it. Correct composes a note in the owner's words and lets the pipeline re-derive.
