Why this design · outcome mapping

The original card stored only {entity_ids, summary} — no names, no facts, no source — so the owner had to leave the app to rule. This screen hydrates the card into a fact ledger: the two entity names head two columns, and one aligned row per shared predicate puts the values side by side. Differing rows carry an amber hairline + DIFFERS dot; matching rows are muted. Seeing "JUL 2" vs "JUL 27" adjacently makes "are these the same thing?" answerable at a glance.

Per-row inputs → one decision

Tap a value to elect a survivor (that side supersedes), or tap Both valid · distinct to declare the row a legit difference. The bottom bar aggregates every row live:

DismissAll rows marked distinct → these are separate appointments. Backend: close card, not_a_contradiction, no wiki change.
Keep A / Keep BAll rows elect one side → the other is a stale duplicate. Backend: supersede — merge/retire the losing entity.
File correctionMixed picks, or "both wrong" → the truth isn't a clean pick. Opens a composer; backend files a correction note (channel #7, the only human-write path — the wiki is never hand-edited).
